  <name>Task 1: Seed artists and tracks from parsed library into PostgreSQL</name>
  <files>pipeline/seed_library.py</files>
  <action>
Create `pipeline/seed_library.py` that:

1. Imports `parse_liked_tracks()` from `parse_library` to get the 9,115 tracks with artist names and spotify track IDs.

2. Connects to PostgreSQL using the same `build_sync_db_url()` pattern from `seed_countries.py` (psycopg2 sync, env vars for POSTGRES_HOST/USER/PASSWORD/DB).

3. Seeds artists first:
   - Collect unique artist names from parsed tracks.
   - For each unique artist: INSERT into `artists` table with `name` only. `spotify_id` will be NULL initially (resolved in Task 2 via Spotify API). Use `INSERT ... ON CONFLICT (name) DO NOTHING` — but note `name` is NOT unique in the schema. Instead, check if artist exists by name before inserting (SELECT first, INSERT if not found). Or better: maintain a local dict of artist_name -> artist_id, insert each unique artist, track the returned ID.
   - Actually, the artists table has `spotify_id` as UNIQUE but `name` is not unique. The correct approach: INSERT each unique artist by name (no conflict clause on name since it's not unique — just do a plain INSERT). Before inserting, SELECT to check if an artist with that name already exists (for idempotency on re-run). Store artist_name -> artist_id mapping.
   - Use a transaction: insert all artists, then commit once.

4. Seeds tracks:
   - For each parsed track: INSERT into `tracks` table with `name`, `spotify_id` (track spotify ID from URI), `artist_id` (looked up from the artist_name -> artist_id mapping), `album_name`.
   - Use `INSERT ... ON CONFLICT (spotify_id) DO NOTHING` since tracks.spotify_id is UNIQUE NOT NULL.
   - Commit in batches of 500 for efficiency.

5. Seeds user_tracks:
   - For each track, also INSERT into `user_tracks` with `track_id` and `artist_id`.
   - Use INSERT with conflict avoidance if re-running.

6. Prints summary: "Seeded X artists, Y tracks, Z user_tracks".

7. Add CLI entrypoint with `--env-file` arg (same pattern as seed_countries.py).

8. Add `--export-path` arg to specify YourLibrary.json location (defaults to `~/Downloads/Spotify Account Data/YourLibrary.json`).

IMPORTANT: The Spotify export does NOT contain artist Spotify IDs — only track URIs. Artist spotify_id must be resolved via the Spotify API in Task 2. Seed artists with `spotify_id = NULL` initially.

Follow the exact psycopg2 pattern from seed_countries.py (manual conn/cursor, raw SQL, explicit commit). Do NOT use SQLAlchemy ORM for pipeline scripts.

<task type="auto">
  <name>Task 2: Spotify artist metadata enrichment with batch fetch</name>
  <files>pipeline/enrich_spotify.py, pipeline/requirements.txt</files>
  <action>
Create `pipeline/enrich_spotify.py` that:

1. Reads the audio features flag file (`pipeline/.audio_features_available`). If AUDIO_FEATURES_AVAILABLE=false (expected), log "Audio features unavailable — skipping audio feature enrichment" and proceed to artist metadata only. Do NOT call `sp.audio_features()` under any circumstance.

2. Connects to PostgreSQL (same `build_sync_db_url()` pattern).

3. Resolves Spotify artist IDs for all artists with `spotify_id IS NULL`:
   - For each artist, search Spotify by name using `sp.search(q=artist_name, type='artist', limit=1)`.
   - Take the top result. Compare the result name (normalized: lowered, stripped) against the search query. If match, extract the Spotify artist ID and UPDATE the artists row.
   - If no match or no results, skip (leave spotify_id NULL). Log the skip.
   - Commit after each artist (checkpoint pattern — safe to resume if interrupted).
   - This step takes ~3,022 API calls at ~2-3 per second (Spotify rate limit is higher than MusicBrainz). Log progress every 100 artists.

4. Fetches artist metadata in batches of 50:
   - Query all artists WHERE `spotify_id IS NOT NULL AND genres IS NULL` (unprocessed artists).
   - Batch the spotify_ids into groups of 50.
   - Call `sp.artists(batch)` for each batch. The Spotify `artists()` endpoint returns up to 50 artists per call. Do NOT pass more than 50 IDs.
   - For each returned artist: extract `genres` (list, may be empty), `popularity` (int), `image_url` (first image URL from images array, or NULL).
   - UPDATE the artists row: SET genres, popularity, image_url, updated_at = NOW().
   - Commit after each batch.

5. Uses `spotipy.Spotify(auth_manager=SpotifyClientCredentials(...))` with env vars SPOTIFY_CLIENT_ID and SPOTIFY_CLIENT_SECRET.

6. Prints summary at end: "Resolved X/Y artist IDs, enriched Z artists with metadata, W had empty genres".

7. CLI entrypoint with `--env-file` arg.

`pipeline/requirements.txt` — no changes needed (spotipy already listed). Verify musicbrainzngs is NOT added yet (that's 02-02).

IMPORTANT: The `genres` column is PostgreSQL `text[]` (ARRAY). When writing with psycopg2, pass a Python list directly — psycopg2 handles the array adaptation. Example: `cur.execute("UPDATE artists SET genres = %s WHERE id = %s", ([genre1, genre2], artist_id))`.

Handle Spotify API rate limiting gracefully: if a 429 response is received, respect the Retry-After header. spotipy handles this automatically with retries, but log when it happens.
