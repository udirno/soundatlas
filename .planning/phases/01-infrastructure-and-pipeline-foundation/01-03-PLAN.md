  <files>

  <action>
    Create `pipeline/parse_library.py` as an importable module with CLI entrypoint (per user decision: importable for Phase 2 reuse, CLI arg or env var for input path, skip malformed entries with warning, deduplicate by Spotify track ID).

    **Module function** `parse_liked_tracks(export_path: str | Path) -> list[dict]`:
    - Open and parse the JSON file at `export_path`
    - The Spotify data export `YourLibrary.json` has a `"tracks"` key containing a list of track objects
    - IMPORTANT: The exact field names in YourLibrary.json are LOW confidence per research. The script must inspect the actual file structure first. Expected fields based on Spotify export format:
      - `"track"` or `"trackName"` — track name
      - `"artist"` or `"artistName"` — artist name (flat string, not nested array)
      - `"album"` or `"albumName"` — album name
      - `"trackUri"` or `"uri"` — Spotify URI containing the track ID (format: `spotify:track:{id}`)
    - Extract Spotify track ID from URI by splitting on `:` and taking the last segment
    - For each track, produce a dict with keys: `spotify_id`, `name`, `artist_name`, `album_name`
    - Skip entries missing artist name OR missing/invalid Spotify URI — log a warning with the track name if available: `logger.warning(f"Skipping malformed entry: {reason}")`
    - Deduplicate by `spotify_id` — keep first occurrence, log count of duplicates at end
    - Use Python `logging` module (not print) for warnings — configure at INFO level for CLI, let importers configure their own level
    - Return the deduplicated list sorted by artist name (for predictable output)

    **CLI entrypoint** (in `if __name__ == "__main__":` block):
    - Accept `--path` argument via argparse
    - Fall back to `SPOTIFY_EXPORT_PATH` environment variable
    - Fall back to `~/Downloads/Spotify Account Data/YourLibrary.json` as default
    - Call `parse_liked_tracks(path)`
    - Print summary: total tracks parsed, duplicates removed, malformed entries skipped, unique artists count
    - Example output:
      ```
      Parsed 9,115 liked tracks from YourLibrary.json
        Duplicates removed: 23
        Malformed entries skipped: 2
        Unique artists: 3,022
      ```

    IMPORTANT: The parser does NOT write to the database. It returns an in-memory list. Phase 2 handles database insertion.

    Gracefully handle the case where the file structure differs from expected — if the top-level key isn't `"tracks"`, try to detect the correct key by looking for a list of objects with URI-like fields. If truly unrecognizable, raise a clear ValueError with the actual top-level keys found.

<task type="auto">
  <name>Task 2: Spotify audio features endpoint validation script</name>
  <files>
    pipeline/validate_audio_features.py
  </files>
  <action>
    Create `pipeline/validate_audio_features.py` as a standalone validation script (per user decision: separate from parser, uses real track ID from user's export, single track test, writes config flag on 403).

    **Script behavior:**
    1. Load .env from project root using python-dotenv
    2. Read SPOTIFY_CLIENT_ID and SPOTIFY_CLIENT_SECRET from environment
    3. If either is empty/missing, print error and exit with code 1: "SPOTIFY_CLIENT_ID and SPOTIFY_CLIENT_SECRET must be set in .env"
    4. Create a spotipy client using Client Credentials flow:
       ```python
       import spotipy
       from spotipy.oauth2 import SpotifyClientCredentials
       auth_manager = SpotifyClientCredentials(
           client_id=os.environ["SPOTIFY_CLIENT_ID"],
           client_secret=os.environ["SPOTIFY_CLIENT_SECRET"]
       )
       sp = spotipy.Spotify(auth_manager=auth_manager)
       ```
    5. Extract a real track ID from the user's export:
       - Import `parse_liked_tracks` from `parse_library` (importable module from Task 1)
       - Get the first track's `spotify_id`
       - Use `--export-path` CLI arg or `SPOTIFY_EXPORT_PATH` env var or default path for the export file
    6. Call `sp.audio_features([track_id])` wrapped in try/except
    7. Handle outcomes:
       - **Success (data returned):** Print "Audio features endpoint is AVAILABLE" with the returned data for the test track. Write flag file.
       - **403 Forbidden (spotipy.SpotifyException with http_status 403):** Print "Audio features endpoint returned 403 — endpoint restricted for this app (expected for apps registered after Nov 2024)". Write flag file with `false`. This is the EXPECTED outcome per research.
       - **Other error:** Print error details, write flag file with `false`, exit with warning.
    8. Write flag file at `pipeline/.audio_features_available`:
       ```
       # Auto-generated by validate_audio_features.py
       # Do not edit manually
       AUDIO_FEATURES_AVAILABLE=true   # or false
       VALIDATED_AT=2026-03-24T...
       VALIDATED_TRACK_ID=xxx
       ```
       This file is read by Phase 2 enrichment scripts to decide whether to attempt audio features batch fetch.
    9. Also print a human-readable summary of what Phase 2 should do based on the result.

    The script must handle the case where spotipy wraps 403 differently — check both `SpotifyException` and generic HTTP errors. Log the full response for debugging.

    Add `pipeline/.audio_features_available` to `.gitignore` (it contains no secrets but is machine-specific).
  </action>
